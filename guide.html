Building a Claude Code Agent in Gleam

A step-by-step guide to writing your own coding agent — designed from first principles in Gleam, not mechanically translated from Swift.

Based on Ivan Magda’s Swift series. Each stage adds exactly one mechanism to an invariant core loop. Gleam 1.16+ · Erlang target · May 2026

Stage 0

Project Bootstrap

Goal

Scaffold a Gleam project and add the dependencies we’ll need: an HTTP client for the Anthropic API and a file-system library for tool implementations.

Not a Swift Translation

The Swift project splits into two SPM targets (Core library + cli executable) because SPM executables can’t be imported by tests. Gleam has no such limitation — every module under src/ is equally importable. We don’t need a Core/cli split. We just need modules.

$ gleam new claude_code && cd claude_code

This creates src/claude_code.gleam (the entry point) and test/claude_code_test.gleam. By Gleam convention, the module matching the project name is the application root.

Stage 1

API Client & Agent Loop

Goal

Write the HTTP client for the Anthropic Messages API, define the API types as Gleam custom types, and build the core agent loop: send a user query with a bash tool, execute tool calls, feed results back, loop until the model responds with text. Add a REPL so the user can type queries interactively.

Key Gleam Design Decisions

  • No classes. The “agent” is a recursive function that threads immutable state as parameters. Each iteration builds a new state record from the previous one.
  • No async/await. Gleam runs on the Erlang VM, where I/O blocks the calling process (which is fine — Erlang processes are cheap). The agent loop is a tail-recursive function; the BEAM compiler optimises tail calls into jumps.
  • Custom types for API models. Gleam’s algebraic data types model the API’s polymorphic content blocks precisely — each variant carries exactly the fields it needs, with compiler-guaranteed exhaustive pattern matching.
  • Result for all fallible operations. No exceptions, no try/catch. Every HTTP call, file read, and shell command returns Result(a, e).

pub fn create_message(
  client: Client,
  model: String,
  system_prompt: String,
  messages: List(Message),
  tools: List(ToolDef),
  max_tokens: Int,
) -> Result(APIResponse, String) {
  let body = json.object([
    #("model", json.string(model)),
    #("max_tokens", json.int(max_tokens)),
    #("system", json.string(system_prompt)),
    #("messages", encode_messages(messages)),
    #("tools", encode_tools(tools)),
  ]) |> json.to_string

  let req = request.new()
    |> request.set_method(Post)
    |> request.set_header("content-type", "application/json")
    |> request.set_header("x-api-key", client.api_key)
    |> request.set_header("anthropic-version", "2023-06-01")
    |> request.set_body(body)
    |> request.set_url(client.base_url <> "/v1/messages")

  use resp <- result.try(httpc.send(req) |> map_err)
  decode_response(response.body(resp))
}

src/claude_code/agent.gleam — the loop
import claude_code/api.{type Client, type ContentBlock, type Message,
  type StopReason, type ToolDef}
import claude_code/api
import claude_code/internal/shell

pub type AgentState {
  AgentState(
    client: Client, model: String, system_prompt: String,
    messages: List(Message), cwd: String,
  )
}

// Sends a user query through the agent loop. Returns the final text response.
pub fn run(state: AgentState, query: String) -> Result(#(String, AgentState), String) {
  let messages = append_user_message(state.messages, query)
  let state = AgentState(..state, messages: messages)
  agent_loop(state, 0)
}

// The invariant: API call → check stop_reason → execute tools → append results → recurse
fn agent_loop(state: AgentState, iteration: Int) -> Result(#(String, AgentState), String) {
  use response <- result.try(
    api.create_message(state.client, state.model, state.system_prompt, state.messages, tool_defs(), 4096)
  )
  let state = append_assistant(state, response.content)

  case response.stop_reason {
    ToolUse -> {
      use results <- result.try(process_tool_uses(response.content, state))
      let state = append_user(state, results)
      agent_loop(state, iteration + 1)
    }
    _ -> Ok(#(extract_text(response.content), state))
  }
}

// Tool dispatch: one case branch per tool — exhaustive, compiler-checked
fn execute_tool(name: String, input: json.Json, state: AgentState) -> Result(ContentBlock, String) {
  case name {
    "bash" -> execute_bash(input, state.cwd)
    other ->
      Ok(ToolResult(..make_error_result(), content: "Unknown tool: " <> other, is_error: True))
  }
}

fn execute_bash(input: json.Json, cwd: String) -> Result(ContentBlock, String) {
  use cmd <- result.try(get_string(input, "command"))
  use output <- result.try(shell.execute(cmd, cwd, 30))
  Ok(tool_ok(output))
}

fn tool_defs() -> List(ToolDef) {
  [ ToolDef(name: "bash", description: "Run a shell command", input_schema: bash_schema()) ]
}
src/claude_code.gleam — REPL entry point
import claude_code/agent.{type AgentState}
import claude_code/agent
import claude_code/api.{type Client}
import gleam/io
import gleam/string

pub fn main() { let _ = repl(initial_state()) }

fn initial_state() -> AgentState {
  let api_key = get_env("ANTHROPIC_API_KEY")
  let model = get_env("MODEL_ID") |> string.default("claude-sonnet-4-6")
  AgentState(
    client: Client(api_key:, base_url: "https://api.anthropic.com"),
    model:, system_prompt: system_prompt(), messages: [], cwd: current_dir(),
  )
}

fn repl(state: AgentState) -> Nil {
  io.print("> ")
  case io.read_line() {
    Error(_) -> Nil  // EOF
    Ok(input) -> {
      case string.trim(input) {
        "" -> repl(state)
        "exit" | "quit" -> Nil
        query -> {
          case agent.run(state, query) {
            Ok(#(response, new_state)) -> {
              io.println(response)
              repl(new_state)
            }
            Error(e) -> { io.println("Error: " <> e); repl(state) }
          }
        }
      }
    }
  }
}

The AgentState record captures everything the loop needs. run returns Result(#(String, AgentState), String) — a tuple of the response text and the new state with updated messages. The REPL passes this state forward into the next turn, preserving conversation history. No mutation, no hidden side effects.

Stage 2

Tool Dispatch & File Tools

Goal

Add read_file, write_file, and edit_file tools. Build a path sandbox that prevents the agent from escaping the workspace directory. The dispatch mechanism grows from a single case branch to five.

Gleam Dispatch Pattern

Swift uses a [String: handler] dictionary of closures. Gleam doesn’t have heterogeneous closure collections. The idiomatic approach is a case expression on the tool name string. Each tool is one branch. Adding a tool is one new branch. The compiler warns on non-exhaustive patterns if we forget a tool. No dictionary, no registration, no protocol — just pattern matching.

New files this stage

We extract the sandbox into its own module so every file-using tool (now and in future stages) shares the same safety check. The file operation handlers are pure functions that take json.Json input and return Result(ContentBlock, String).

src/claude_code/internal/sandbox.gleam
import simplifile
import gleam/string

pub fn resolve(relative: String, cwd: String) -> Result(String, String) {
  let candidate = case string.starts_with(relative, "/") {
    True -> relative
    False -> cwd <> "/" <> relative
  }
  use resolved <- result.try(
    simplifile.absolute(candidate) |> result.replace_error("Cannot resolve path")
  )
  case string.starts_with(resolved, cwd) {
    True -> Ok(resolved)
    False -> Error("Path escapes workspace: " <> relative)
  }
}
src/claude_code/internal/file_ops.gleam
import claude_code/internal/sandbox
import gleam/json
import gleam/string
import simplifile

// Returns the first occurrence index, or Error if old_text is not in content.
pub fn read_file(input: json.Json, cwd: String) -> Result(String, String) {
  use path <- result.try(require_string(input, "path"))
  use resolved <- result.try(sandbox.resolve(path, cwd))
  use content <- result.try(simplifile.read(resolved))
  let limit = input
    |> get_int("limit")
    |> result.unwrap(2000)
  Ok(truncate_lines(content, limit, 50_000))
}

pub fn write_file(input: json.Json, cwd: String) -> Result(String, String) {
  use path <- result.try(require_string(input, "path"))
  use content <- result.try(require_string(input, "content"))
  use resolved <- result.try(sandbox.resolve(path, cwd))
  use _ <- result.try(simplifile.create_directory_all(parent(resolved)))
  use _ <- result.try(simplifile.write(resolved, content))
  Ok("Wrote " <> int.to_string(string.byte_size(content)) <> " bytes to " <> path)
}

pub fn edit_file(input: json.Json, cwd: String) -> Result(String, String) {
  use path <- result.try(require_string(input, "path"))
  use old_text <- result.try(require_string(input, "old_text"))
  use new_text <- result.try(require_string(input, "new_text"))
  use resolved <- result.try(sandbox.resolve(path, cwd))
  use content <- result.try(simplifile.read(resolved))
  use edited <- result.try(replace_once(content, old_text, new_text, path))
  use _ <- result.try(simplifile.write(resolved, edited))
  Ok("Edited " <> path)
}

The edit_file handler replaces only the first occurrence of old_text (matching Claude Code’s real tool behaviour). Single-occurrence replacement forces the model to be precise about what it wants to change. The require_string helper extracts a field from the JSON input or returns Error("Missing parameter: ...") — keeping the handler bodies clean.

Stage 3

Todo Tool

Goal

Give the agent a structured notepad. The todo list stays visible near the end of the context window as a tool result, fighting instruction-following decay. Add a nag that reminds the agent to update its plan after N turns of inactivity.

Pure Functional State

Swift uses a class TodoManager with mutable items. In Gleam, the todo list is a List(TodoItem) field on AgentState. Every change produces a new AgentState record via the record update syntax AgentState(..old, todos: new_todos). No mutation, no shared references, trivially testable.

New file this stage

internal/todo.gleam ← TodoItem, validation, rendering
src/claude_code/internal/todo.gleam
pub type TodoStatus { Pending; InProgress; Completed }

pub type TodoItem {
  TodoItem(id: String, text: String, status: TodoStatus)
}

pub const max_items = 20

// Validates and replaces the entire list (no incremental diffs)
pub fn validate(items: List(TodoItem)) -> Result(List(TodoItem), String) {
  case list.length(items) > max_items {
    True -> Error("Too many items (max 20)")
    False -> case list.filter(items, in_progress) |> list.length > 1 {
      True -> Error("Only one task may be in-progress at a time")
      False -> Ok(items)
    }
  }
}

pub fn render(items: List(TodoItem)) -> String {
  case items {
    [] -> "No todos."
    _ -> {
      let lines = list.map(items, fn(i) { marker(i.status) <> " " <> i.text })
      let done = items |> list.filter(fn(i) { i.status == Completed }) |> list.length
      string.join(lines, "\n") <> "\n(" <> int.to_string(done) <> "/"
        <> int.to_string(list.length(items)) <> " completed)"
    }
  }
}

fn marker(s: TodoStatus) -> String {
  case s { Pending -> "[ ]"; InProgress -> "[>]"; Completed -> "[x]" }
}

Changes to AgentState and the loop

AgentState gains two fields: todos: List(TodoItem) and turns_without_todo: Int. The nag fires when turns_without_todo >= 3 and the todo list has open items — a short text block "Update your todos." is appended after tool results. The counter resets to 0 on any todo tool call.

This is the first stage where AgentState grows beyond what run needs. The record grows fields as the system grows capabilities — no inheritance, no subclasses, just data.

Stage 4

Subagents

Goal

Let the agent spawn subagents for research tasks. A subagent gets a clean message array, restricted tools, and an iteration cap. Its intermediate history is discarded — only the final text returns to the parent, keeping the parent’s context clean.

Parameterisation, Not Inheritance

Swift extracts the loop into a method and returns a tuple (text, messages). In Gleam, the loop already takes state as a parameter. A subagent calls agent_loop with a fresh AgentState (empty messages, restricted tool list) and discards the returned messages. The parent calls it with its accumulated messages and keeps the returned messages. Same function, different usage — pure parameterisation.

The subagent config uses an allowlist of tool names. The agent tool is excluded (preventing unbounded recursion), as is todo (the subagent shouldn’t maintain its own plan). The 30-iteration cap prevents runaway subagents. The agent_loop function receives the config and filters tool definitions accordingly before each API call.

Subagent handler in agent.gleam
fn execute_subagent(input: json.Json, state: AgentState) -> Result(String, String) {
  use prompt <- result.try(require_string(input, "prompt"))
  // Fresh state: clean messages, restricted config
  let sub_state = AgentState(..state,
    messages: [user_msg(prompt)],
    config: config_subagent(),
    todos: [], turns_without_todo: 0,
  )
  use result <- result.try(agent_loop(sub_state, 0))
  let #(text, _discarded_state) = result
  // Subagent's messages are intentionally dropped — context isolation
  Ok(truncate(text, 10_000))
}

The discarded _discarded_state binding makes the intent explicit: the subagent’s conversation history is thrown away. Only the final text comes back. The parent’s context doesn’t bloat with the subagent’s intermediate file reads and shell output.

Stage 5

Skill Loading

Goal

Add a two-layer skill system. Layer 1 embeds one-line descriptions in the system prompt (~100 tokens total). Layer 2 loads the full skill body only when the model explicitly calls load_skill — knowledge arrives on demand, costing tokens only when needed.

Startup-Time Loading, Immutable Lookup

Swift parses YAML frontmatter character-by-character. Gleam does the same with string scanning — no YAML library needed. The SkillLoader is a record with a Dict(String, Skill) built at startup. All lookups are pure dictionary reads. The system prompt builder receives the descriptions string as a parameter; an empty string means no skills are configured (graceful degradation).

src/claude_code/internal/skill_loader.gleam (key parts)
pub type Skill {
  Skill(name: String, description: String, body: String)
}

pub type SkillLoader {
  SkillLoader(skills: Dict(String, Skill))
}

// Scans skills/{name}/SKILL.md for each subdirectory
pub fn load(directory: String) -> SkillLoader {
  case simplifile.is_directory(directory) {
    False -> SkillLoader(skills: dict.new())
    True -> {
      let entries = simplifile.read_directory(directory) |> result.unwrap([])
      let skills = entries
        |> list.filter_map(try_load(directory, _))
      SkillLoader(skills: dict.from_list(skills))
    }
  }
}

// Cheap layer: ~1 line per skill (~100 tokens total)
pub fn menu(loader: SkillLoader) -> String {
  case dict.is_empty(loader.skills) {
    True -> ""
    False -> loader.skills
      |> dict.values
      |> list.sort(fn(a, b) { string.compare(a.name, b.name) == Lt })
      |> list.map(fn(s) { "  - " <> s.name <> ": " <> s.description })
      |> string.join("\n")
  }
}

// Expensive layer: full body, only on demand
pub fn body_for(loader: SkillLoader, name: String) -> String {
  case dict.get(loader.skills, name) {
    Ok(skill) -> " <> name <> "\">\n" <> skill.body <> "\n"
    Error(_) -> "Unknown skill '" <> name <> "'"
  }
}

Skills live in skills/{name}/SKILL.md with YAML frontmatter for name and description. The frontmatter parser is a simple line-by-line scan between --- delimiters. Subagents automatically inherit load_skill — the allowlist only excludes agent and todo.

Stage 6

Context Compaction

Goal

Build three compression layers so the agent can run indefinitely. Micro-compact replaces old tool results with placeholders. Auto-compact triggers LLM summarisation when estimated tokens cross a threshold. The compact tool lets the model request summarisation itself.

Pure Data Transformation

All three compaction layers are pure functions: List(Message) → List(Message). Token estimation uses byte_size(encoded_json) / 4 as a heuristic — no tokenizer dependency. Micro-compact runs before every API call (structural pruning). Auto-compact checks the threshold after micro-compact. Manual compact fires after tool results are appended. On failure, compaction returns the original messages unchanged — compaction must never crash the loop.

Micro-compact scans message content for ToolResult blocks, keeps the 3 most recent, and replaces older ones with "[Previous: used bash]" placeholders. Results shorter than 100 characters are left alone. Auto-compact saves the full transcript as JSONL to .transcripts/, asks the LLM to produce a three-part summary (accomplishments, current state, key decisions), and returns a compressed 2-message conversation.

Stage 7

Task System (DAG)

Goal

Build a file-based DAG task store so the agent’s plan survives context compaction and process restarts. Each task is a standalone JSON file in .tasks/. Dependencies form a directed acyclic graph via blocked_by / blocks arrays. Auto-incrementing IDs survive restarts by scanning the directory.

Filesystem as Database

Swift uses FileManager + JSONEncoder/Decoder. Gleam uses simplifile + gleam/json. Each task is one JSON file — atomic reads and writes, no partial corruption across files. The cascade unblock (when a task completes, remove its ID from every dependent’s blocked_by) is a list.fold over the directory listing. The TaskStore record holds the directory path and the next available ID.

pub type TaskStatus { Pending; InProgress; Completed }

pub type AgentTask {
  AgentTask(
    id: Int, subject: String, description: String,
    status: TaskStatus, blocked_by: List(Int), blocks: List(Int),
  )
}

pub type TaskStore { TaskStore(directory: String, next_id: Int) }

pub fn init(directory: String) -> TaskStore {
  let _ = simplifile.create_directory_all(directory)
  TaskStore(directory:, next_id: scan_max_id(directory) + 1)
}

// Create returns the new store (with incremented next_id) and the task JSON
pub fn create(
  store: TaskStore, subject: String, description: String,
) -> Result(#(TaskStore, String), String) {
  let task = AgentTask(id: store.next_id, subject:, description:,
    status: Pending, blocked_by: [], blocks: [])
  use _ <- result.try(write_task(store.directory, task))
  let store = TaskStore(..store, next_id: store.next_id + 1)
  Ok(#(store, task_to_json(task)))
}

// When a task completes, scan all files and remove its ID from blocked_by arrays
fn cascade_unblock(directory: String, completed_id: Int) -> Nil {
  use files <- result.unwrap(simplifile.read_directory(directory), [])
  let _ = list.each(files, fn(file) {
    case read_task(directory, extract_id(file)) {
      Ok(task) if list.contains(task.blocked_by, completed_id) -> {
        let updated = AgentTask(..task,
          blocked_by: list.filter(task.blocked_by, fn(id) { id != completed_id }))
        let _ = write_task(directory, updated)
      }
      _ -> Nil
    }
  })
  Nil
}

Four new tool handlers: task_create, task_update, task_list, task_get. The subagent allowlist excludes task_create and task_update — subagents get read-only task access. AgentState gains a task_store: TaskStore field, threaded through the loop alongside messages and todos.

Stage 8

Background Tasks

Goal

Run slow commands (builds, test suites) in the background while the agent continues working. Results drain as notifications into the agent loop before each API call. The single mutable-state holder in the system is a dedicated OTP actor.

OTP Actor, Not Swift Actor

Swift uses a single actor keyword for thread-safe state. Gleam achieves the same with gleam/otp/actor — the Erlang OTP gen_server pattern. The background manager is a long-lived process holding a job registry and notification queue. Communication is explicit message passing: Run to start a job, DrainNotifications to collect completed results. No locks, no race conditions, no shared memory. This is the only place in the system where mutable state lives behind a process boundary. Everything else is pure data flow.

import gleam/erlang/process.{type Subject}
import gleam/otp/actor
import claude_code/internal/shell

pub type JobStatus { Running; Completed; Error; Timeout }

pub type Notification {
  Notification(job_id: String, status: JobStatus, result: String)
}

// Messages the actor understands
pub type Message {
  RunJob(command: String, timeout_s: Int, reply: Subject(String))
  Drain(reply: Subject(List(Notification)))
}

pub fn start(cwd: String) -> Subject(Message) {
  let initial = State(jobs: dict.new(), notifications: [], cwd:)

  actor.new(initial, fn(state: State, msg: Message) {
    case msg {
      RunJob(command:, timeout_s:, reply:) -> {
        let job_id = generate_id()
        let job = Job(id: job_id, command:, status: Running)
        // Spawn a separate process to run the command
        let _ = process.spawn(fn() { run_and_notify(job_id, command, timeout_s, state.cwd, actor_self) })
        process.send(reply, job_id)
        actor.Continue(State(..state, jobs: dict.insert(state.jobs, job_id, job)))
      }
      Drain(reply:) -> {
        process.send(reply, state.notifications)
        actor.Continue(State(..state, notifications: []))
      }
    }
  })
  |> actor.start
  |> result.unwrap_both
}

Loop integration

Before each API call, the loop drains notifications from the background actor and appends them to the message list as <background-results> XML blocks. If the last message is already from the User role, results append to it; otherwise a new user message is created. A synthetic assistant acknowledgment ("Noted background results.") satisfies the user→assistant turn alternation requirement. LoopConfig gains a drain_background: Bool field — subagents set it to False to prevent consuming the parent’s notifications.

Two new tool handlers: background_run (sends RunJob to the actor, returns immediately with a job ID) and background_check (returns the current job list from the actor). The agent loop can now fire off a build in the background and continue working while it runs.

What Makes This “The Gleam Way”

  • AgentState is a record, not an object. Every “method” is a function that takes state and returns a new state. The record update syntax AgentState(..old, field: new_value) makes evolution explicit.
  • The loop is a tail-recursive function. No while true, no mutable loop variables. Each iteration builds the next iteration’s arguments and recurses. The BEAM optimises tail calls into jumps.
  • Tool dispatch is a case expression. No dictionary of closures, no protocol registry. One branch per tool. Adding a tool is one new branch. The compiler checks exhaustiveness.
  • Result is the universal error type. Every fallible operation returns Result(a, e). Chaining uses use expressions and result.try. Error types are plain strings that compose naturally.
  • One OTP actor for the one mutable spot. Background job state lives behind a process boundary. Everything else — messages, todos, tasks, config — is immutable data threaded through function parameters.
  • Filesystem as durable memory. Tasks are JSON files. Transcripts are JSONL files. Skills are markdown files. The filesystem is the database; simplifile is the query engine.
  • Every stage adds one thing. The loop shape from Stage 1 (API call → check stop_reason → execute tools → append results → recurse) hasn’t changed. Tools are the variable; the loop is the invariant.
